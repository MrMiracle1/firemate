## 一、架构问题 (Critical Issues)

| 问题 | 严重程度 | 位置 | 描述 |

| 无请求体验证 | 🔴 严重 | 所有 routes | 所有API端点直接接受客户端数据，没有输入验证。SQL注入风险（虽然用Supabase但仍需验证）|

| 敏感信息泄露 | 🟡 中等 | `server/src/index.ts:39` | 错误处理直接返回 `err.message`，可能泄露内部实现细节 |

**建议**: 添加 JWT 认证中间件，输入验证中间件 (如 `express-validator`)

### 1.2 后端数据一致性

| 问题 | 严重程度 | 位置 | 描述 |
|------|---------|------|------|
| 余额更新非原子操作 | 🔴 严重 | `transactionService.ts:134-191` | 余额更新分两步：先读取再更新。并发请求会导致数据丢失 |
| 无数据库事务 | 🔴 严重 | `transactionService.ts` | 创建/更新交易时，余额更新失败不会回滚主操作 |

**示例问题代码**:
```typescript
// 读取余额
const { data } = await supabase.from('accounts').select('balance').eq('id', accountId).single();
// 更新余额 - 如果这两步之间有另一个请求，余额会出错
await supabase.from('accounts').update({ balance: Number(data.balance) + amount }).eq('id', accountId);
```

**建议**: 使用 PostgreSQL 事务或数据库函数 (stored procedure) 来保证原子性
