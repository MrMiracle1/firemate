# 第一部分：产品愿景与设计理念

## 一、为什么做这个产品

### 1.1 核心痛点

我们每个人手机里可能同时装着：银行卡App、支付宝、微信钱包、股票软件、基金App... 钱散落在各处，就像散落在地上的硬币，捡不起来，也看不清全貌。

**用户最真实的困扰：**

| 痛点 | 场景描述 |
|------|----------|
| 钱太散 | 不知道微信里还剩多少钱，支付宝又花了多少 |
| 记流水太累 | 每次消费完要打开App选分类、输入金额，步骤太繁琐 |
| 不知道钱花哪了 | 月底一看账单，发现什么都没剩下，但想不起来钱花哪了 |
| 存不下钱 | 工资到账后不知不觉就花光了，没有规划 |

### 1.2 产品要解决的问题

这款财务管理App的核心使命：**让用户对自己的钱有足够的掌控力**

我们要做到：
1. **一个界面看清所有钱** - 把散落各处的账户汇总到一起
2. **记账尽可能简单** - 几步完成，不打断用户的消费节奏
3. **知道钱去了哪里** - 分类统计，清晰了解消费结构
4. **能存下钱** - 通过预算控制和目标储蓄，帮助用户养成储蓄习惯

---

## 二、为谁设计

### 2.1 核心用户画像

**小明的故事（典型用户）**

> 小明，28岁，工作3年，月薪1.5万。
>
> 他的钱散落在：
> - 工资卡（建行）
> - 微信钱包（日常消费）
> - 支付宝（网购）
> - 股票账户（被套）
> - 余额宝（存了一点）
>
> 他现在的困扰：
> - 不知道现在到底有多少钱
> - 每次花钱就记一笔太麻烦了，经常忘记
> - 月底发现钱没了，但想不起来花在哪
> - 想存钱但总是月光

### 2.2 用户需求层次

```
┌─────────────────────────────────────────┐
│            高级需求（V2.0+）            │
│    AI智能分析、资产增长可视化、语音记账   │
├─────────────────────────────────────────┤
│            中级需求（V1.0）              │
│       预算控制、目标储蓄、报表分析        │
├─────────────────────────────────────────┤
│            基础需求（V1.0）              │
│      账户管理、记账、查看资产总览         │
└─────────────────────────────────────────┘
```

**V1.0版本解决的是最基础的"能看到钱、能记流水"的需求，这是整个产品的地基。**

# 第二部分：设计逻辑与设计故事

## 三、核心设计理念

### 3.1 设计原则

| 原则 | 含义 | 落地方式 |
|------|------|----------|
| **先简单再丰富** | 第一版只做核心功能，确保可用 | V1.0聚焦记账基础功能，复杂功能后续迭代 |
| **所见即所得** | 用户看到的数字就是真实的 | 余额实时计算，所见即所管 |
| **渐进式引导** | 不强迫用户做复杂操作 | 预算可设可不设，分类可简可繁 |
| **数据可追溯** | 每一笔账都有迹可循 | 保留完整流水，支持查看和删除 |

### 3.2 账户管理 - 设计逻辑

**为什么要先做账户管理？**

> 就像你要管理一个公司，首先要知道公司有哪些部门。
> 账户就是"财务部门"，是所有功能的基础。

**设计故事：**
- 用户第一次打开App，就像走进一个新家
- 首先需要把"家具"搬进来——添加账户
- 账户就是家具：银行卡是桌子，微信是椅子，现金是茶几
- 有了家具，才能开始生活——开始记账

**账户类型的设计思考：**

| 类型 | 存在的理由 |
|------|------------|
| 现金 | 很多人还保留现金习惯，尤其是老人 |
| 银行卡 | 工资卡、储蓄卡，这是"正房" |
| 第三方支付 | 年轻人的日常，微信+支付宝 |
| 投资账户 | 股票、基金，这些也是资产的一部分 |
| 储蓄账户 | 定期、理财，"养鹅基金" |

**核心操作：新增 → 初始余额就是第一笔"钱"**
- 用户创建一个账户，输入"当前有多少钱"
- 系统自动记录为一笔流水，作为资产的"起点"

---

### 3.3 手动记账 - 设计逻辑

**为什么先做手动记账，不做语音记账？**

> 语音记账听起来很酷，但技术难度大，用户习惯培养需要时间。
> 手动记账虽然"原始"，但它是所有记账方式的地基：
> - 手动记好了，语音才能识别得更准
> - 手动记好了，分类和账户体系才能验证

**记账流程的设计：**
```
选择类型 → 选择分类 → 输入金额 → 选择账户 → 选择时间 → 备注 → 确认
```

每一步都很清晰，用户不会迷路。

**分类的设计逻辑：**
- 支出分类：按消费场景分（餐饮、交通、购物...）
- 收入分类：按来源分（工资、兼职、投资...）
- 为什么不搞几十个分类？**够用就行，太多反而选择困难**

**记账页面的位置：**
- 首页正中央的"记一笔"按钮，点开就记账
- 为什么放这么显眼？**降低记账门槛，养成习惯**

---

### 3.4 内部转账 - 设计逻辑

**为什么转账不算收支？**

> 这是一个容易被忽视但很重要的问题：
> - 工资1万块到银行卡 → 收入
> - 转1000到微信钱包 → 转账（不是收入！）
> - 微信花了50买零食 → 支出

如果把转账算成收入/支出，统计报表就会失真。**转账只是钱从一个口袋换到另一个口袋，口袋总数没变。**

---

### 3.5 首页仪表盘 - 设计逻辑

**用户打开App第一眼应该看到什么？**

> 答案是：**我最有钱的时候（总资产）**

**首页三大件：**

| 信息 | 用户心理 |
|------|----------|
| 总资产 | "我现在到底有多少钱？" |
| 本月收支 | "这个月花了多少？还剩多少？" |
| 预算进度 | "这个月还能花多少？" |

**设计逻辑：**
- 总资产卡片放在最上面，最大最醒目 - 满足"看看自己有多少钱"的冲动
- 本月收支紧跟其后 - 满足"这个月花了多少"的好奇
- 预算进度是"控制阀" - 满足"别超支"的理性需求

---

### 3.6 预算管理 - 设计逻辑

**预算不是限制，而是安全感**

> 很多人觉得预算就是"限制自己花钱"，这是误解。
> 预算的本质是：**给你一个安全边界，让你花得安心。**

**V1.0的设计：**
- 只做一个**月度总预算** - 简单，不复杂
- 用进度条展示 - 直观，一眼看懂
- 三色预警：
  - 绿色（0-70%）：安全区，放心花
  - 橙色（70-90%）：警告区，该收敛了
  - 红色（90%+）：危险区，别花了

**为什么不一开始就做分类预算？**
- 用户还没养成记账习惯，设定分类预算意义不大
- V1.0先验证"预算"这个概念是否有效，后续再细化

---

### 3.7 报表可视化 - 设计逻辑

**数据要讲人话**

> 财务报表不是给会计看的，是给普通人看的。
> 普通人只需要知道：**钱花哪了、趋势怎么样**

**V1.0提供三种图：**

| 图表 | 回答的问题 |
|------|------------|
| 饼图 - 支出分类 | "我这个月花钱最多的地方是哪儿？" |
| 折线图 - 月度趋势 | "最近几个月我的收支变化趋势如何？" |
| 条形图 - 账户余额 | "我的钱都分布在哪些账户里？" |

**为什么是这三种？**
- 饼图：占比问题 - 最适合回答"钱花哪了"
- 折线图：趋势问题 - 看变化，看走势
- 条形图：分布问题 - 看各账户的占比

---

### 3.8 目标储蓄 - 设计逻辑

**为什么要有目标？**

> 没有目标的存钱，就像没有终点的跑步，很难坚持。
> 目标就是**给存钱一个理由**。

**V1.0的设计：**
- 设定一个目标金额（比如10万）
- 关联到一个账户（比如"养鹅基金"账户）
- 进度条自动显示：已存金额 / 目标金额

**为什么关联账户？**
- 这样最直观：目标账户余额涨了，进度条就涨了
- 不需要额外操作，存钱进去就自动反映

# 第四部分：功能详细说明

## 五、核心功能逻辑

### 5.1 账户管理功能

#### 5.1.1 支持的账户类型

| 类型 | 子类型 | 余额显示 | 示例 |
|------|--------|----------|------|
| 现金 | - | 显示余额 | 钱包里的现金 |
| 银行卡 | 储蓄卡、信用卡 | 显示余额 | 工资卡、信用卡 |
| 第三方支付 | 支付宝、微信钱包 | 显示余额 | 微信、支付宝 |
| 投资账户 | 股票、基金 | 显示市值 | 证券账户、基金 |
| 储蓄账户 | 定期、理财 | 显示余额 | 定期存款、余额宝 |

#### 5.1.2 账户操作

| 操作 | 说明 |
|------|------|
| 新增账户 | 填写名称、类型、初始余额 |
| 编辑账户 | 可修改名称、颜色、图标、备注 |
| 删除账户 | 删除时提示"账户关联流水将一并删除" |
| 更新余额 | 手动对账，差额自动记录为调账流水 |

#### 5.1.3 转账功能

- 转出账户、转入账户、金额必填
- 转账不计入收支，只改变账户余额
- 转账金额不能超过转出账户余额

---

### 5.2 记账功能

#### 5.2.1 流水类型

| 类型 | 记账方向 | 账户余额变化 |
|------|----------|--------------|
| 收入 | 增加 | +金额 |
| 支出 | 减少 | -金额 |
| 转账 | 不变 | 转出-，转入+ |

#### 5.2.2 分类体系

**支出分类：**
- 餐饮（早餐、午餐、晚餐、零食、外卖）
- 交通（公交、地铁、出租、加油、停车）
- 购物（服装、日用品、电子产品）
- 居住（房租、水电、物业）
- 娱乐（电影、游戏、旅游）
- 医疗（门诊、药品、保险）
- 教育（培训、书籍、课程）
- 其他

**收入分类：**
- 工资（月薪、奖金）
- 兼职（外快、项目收入）
- 投资（股票收益、基金收益）
- 其他（红包、礼金）

---

### 5.3 预算功能

- 月度总预算，金额必填且大于0
- 每月可修改1次当月预算
- 进度显示：
  - 绿色：0-70%
  - 橙色：70-90%
  - 红色：90%+

---

### 5.4 目标储蓄功能

- 目标名称、目标金额、关联账户必填
- 进度自动计算：关联账户余额 / 目标金额
- 支持删除目标

---

## 六、数据计算逻辑

### 6.1 账户余额

```
账户当前余额 = 初始余额 + 收入总和 - 支出总和 + 转入总和 - 转出总和
```

### 6.2 统计数据

| 统计项 | 计算逻辑 |
|--------|----------|
| 总资产 | 所有账户当前余额之和 |
| 本月收入 | 当月1日至今所有收入流水之和 |
| 本月支出 | 当月1日至今所有支出流水之和 |
| 本月结余 | 本月收入 - 本月支出 |
| 预算进度 | 本月支出 / 月度预算 × 100% |
| 目标进度 | 关联账户余额 / 目标金额 × 100% |

---

## 七、边界场景处理

### 7.1 账户

| 场景 | 处理 |
|------|------|
| 删除有关联流水的账户 | 二次确认，删除账户及所有流水 |
| 账户余额为负 | 允许（如信用卡，显示为负债） |
| 修改账户类型 | 不允许，需删除重建 |

### 7.2 记账

| 场景 | 处理 |
|------|------|
| 支出超过账户余额 | 允许，显示警告提示 |
| 删除流水 | 确认后删除，余额自动回滚 |

### 7.3 预算

| 场景 | 处理 |
|------|------|
| 未设置预算 | 首页不显示进度条，显示"未设置" |
| 修改当月预算 | 允许，每月限1次 |
| 预算为0 | 校验不通过，提示输入大于0的金额 |

### 7.4 目标

| 场景 | 处理 |
|------|------|
| 关联账户删除 | 目标失效，提示重新关联 |
| 目标达成 | 显示"已达成"状态 |
